# LLaVA-OneVision — Easy Visual Task Transfer（単一/複数画像/動画の統合）

LLaVA-OneVision は、これまで別々のモデルとして開発されがちだった「単一画像」「複数画像」「動画」という3つの視覚シナリオを、**1つのモデル・1つのレシピ**で扱うオープンな大規模マルチモーダルモデル（LMM）です。LLaVA / LLaVA-1.5 を既習の読者にとって最も腹落ちすべき点は、アーキテクチャ自体は LLaVA 系の最小構成を素直に受け継ぎながら、**視覚表現（AnyRes）と学習データの設計だけ**で3形態を束ね、しかも画像で身につけた能力が動画へ**転移（task transfer）**して創発する、という主張です。本ページではこの「なぜ束ねられ、なぜ転移するのか」を、アーキテクチャ・視覚表現・学習カリキュラムの3点から読み解きます。

## 全体像（まず一枚で）

LLaVA-OneVision の構成は LLaVA 系そのものです。**SigLIP 視覚エンコーダ**が入力をパッチ特徴へ符号化し、**2層 MLP projector** がそれを言語の単語埋め込み空間へ写像し、**Qwen2 LLM** がテキスト指示と並べて自己回帰的に応答を生成します。新しいのはモデル形状ではなく、視覚エンコーダへ入る視覚信号の「形」が、単一画像のクロップ・複数画像中の各画像・動画中の各フレーム、と場面ごとに切り替わる点です。

サイズは **0.5B / 7B / 72B** の3系列が用意され、エッジ用途からクラウドまでをカバーします。設計思想は明快で、「強い LLM が結局はマルチモーダル能力を底上げする」「SigLIP がオープンな視覚エンコーダの中で高い LMM 性能を出す」という LLaVA-NeXT 系の経験則を、そのまま OneVision のレシピへ持ち込んでいます。

## 3つの視覚形態を1つの表現へ（AnyRes）

統合を可能にしている中核が **Higher AnyRes**（バイリニア補間つき AnyRes）です。SigLIP は解像度 $384\times 384$ の入力を **729 トークン**に符号化しますが、これを基本単位として、3形態それぞれに「どう視覚トークンへ展開するか」を変えます。

- **単一画像**：大きな空間構成 $(a, b)$ でタイル分割し、各タイルを符号化したうえでベース画像も加える。視覚トークンを多めに割り当て、**長い系列**で1枚の画像を表現する。これは後述の転移を滑らかにするための意図的な設計で、画像をあえて「動画のような長い系列」に見せる狙いです。
- **複数画像**：ベース解像度のみを符号化し、**多クロップは行わない**。各画像を素直に並べて、画像枚数ぶんのトークンに収めることで計算資源を節約する。
- **動画**：各フレームをベース解像度へリサイズして符号化し、**バイリニア補間でフレームあたりのトークン数を削減**する。これにより「フレーム数を増やす／フレームあたりトークンを減らす」のトレードオフを取り、より多くのフレームを見られるようにする。

ここで効いてくるのが、**3形態の最大視覚トークン数をおおよそ同じ規模に揃える**という設計です。論文の例では、単一画像は $(1+9)\times 729 \approx 7290$、複数画像は $12\times 729 \approx 8748$、動画は $32\times 196 \approx 6272$ と、いずれも数千トークン規模に収まるよう配分されています。この「揃えること」こそが、形態をまたいだ能力転移を成立させるための土台になります。

トークン予算を場面間で揃える効果は、表で見ると掴みやすくなります。

| 視覚形態 | 視覚エンコーダへの入力 | 多クロップ | トークン配分の方針 | 最大トークンの例 |
|---|---|---|---|---|
| 単一画像 | 高解像度タイル＋ベース画像 | あり（大きな $(a,b)$） | 多めに割り当て、長い系列で表現 | $(1+9)\times 729 \approx 7290$ |

## Task Transfer（能力の転移）

LLaVA-OneVision の主張で最も研究的に面白いのが **task transfer による能力創発**です。動画専用の大規模指示データは画像に比べて圧倒的に少ない一方で、LLaVA-OneVision は**画像で学んだ理解能力を動画へ転移**させることで、強い動画理解を獲得します。論文の議論では、より強い「単一画像モデル」ほど、複数画像・動画タスクへうまく転移できる、という関係が繰り返し示されます。

ここで前節の設計が効きます。単一画像をあえて**長い視覚トークン系列**で表現し、動画のフレーム列に近い見え方にしておくこと、そして3形態のトークン予算を揃えておくこと。この2つによって、LLM から見たときの「視覚信号の系列としての姿」が形態をまたいで連続的になり、画像で培った系列処理の能力が動画へ滑らかに引き継がれます。動画を時間方向に並ぶ画像列として扱う AnyRes の素性が、そのまま転移の経路になっているわけです。

## 学習カリキュラム

LLaVA-OneVision は **curriculum learning（易しい例から難しい例へ段階的に）** の原則で学習されます。機能を3つに分解し、段階を分けることで、各段階のチェックポイントを再利用しやすくする狙いもあります。

1. **Stage-1：Language-Image Alignment（言語-画像整列）**。視覚特徴を LLM の単語埋め込み空間へ整列させる段階。**更新するのは projector のみ**で、視覚エンコーダと LLM は据え置きます。ベース画像表現は 729 トークン。
2. **Stage-1.5：High-Quality Knowledge Learning（高品質知識学習）**。固定計算予算のもとで、合成キャプション・Document/OCR・多言語などの高品質データを継続的に注入し、事前学習済み LLM/ViT の知識を磨き上げる段階。ここから**モデル全体を更新**し、AnyRes でトークン予算も増やします。
3. **Stage-2：Visual Instruction Tuning（視覚指示チューニング）**。多様な視覚タスクを「望ましい応答」とともに解けるよう、指示データで学習する段階。これがさらに2フェーズに分かれます。
   - **Single-Image Training**：まず大規模な単一画像指示データ（約 3.2M）で、単一画像の指示追従能力を徹底的に固める。
   - **OneVision Training**：続いて**単一画像・複数画像・動画を混ぜた**データ（約 1.6M：複数画像/動画/単一画像のミックス）で学習し、能力を全形態へ拡張する。ここで形態をまたぐ転移と新能力の創発が起こります。

論文は、この OneVision 段階（事後学習）が、LMM に複数画像・動画の理解力を持たせる**最も単純で費用対効果の高い方法**だろう、と位置づけています。なお学習率設定では、**視覚エンコーダの学習率を LLM の約 1/5 に抑える**など、事前学習済み視覚表現を壊さない配慮も入っています。

## まとめと、読解後に答えたい問い

LLaVA-OneVision の本質は、アーキテクチャの新規性ではなく、**視覚表現（Higher AnyRes）と学習カリキュラムの設計**で単一画像・複数画像・動画を1モデルへ束ね、画像から動画への**task transfer**で動画理解を創発させた点にあります。3形態のトークン予算を揃え、単一画像をあえて長い系列で表現する——この一見地味な工夫が転移の経路を作っています。

読解後に自分の言葉で答えたい問い:

- なぜ単一画像をあえて「長い視覚トークン系列」で表現すると、動画への転移が滑らかになるのか。トークン予算を3形態で揃える設計と、どう噛み合っているか。
- AnyRes で単一画像は多クロップする一方、複数画像と動画では多クロップしないのはなぜか。それぞれの計算予算と狙いの違いを説明できるか。
- 学習を3段階（整列→知識→指示）に分け、Stage-1 では projector のみ、以降は全体を更新するのはなぜか。視覚エンコーダの学習率を小さく保つ意図は何か。
- OneVision 段階（事後学習で3形態を混ぜる）が「最も単純で費用対効果が高い」と言えるのはどんな前提があるからか。動画専用の大規模学習と比べて何を節約しているか。
